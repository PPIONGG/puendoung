# 🎯 ภาพรวมระบบเพื่อนด้วง

เอกสารนี้อธิบายเพื่อนด้วงแบบเข้าใจง่าย สำหรับคนที่เพิ่งมาร่วมโปรเจกต์ ไม่จำเป็นต้องมีพื้นฐาน technical

---

## 🏪 ธุรกิจคืออะไร?

**เพื่อนด้วง** คือร้านค้าออนไลน์ที่ขาย "ด้วง" และอุปกรณ์ที่เกี่ยวข้องกับการเลี้ยงด้วง

ลองนึกถึงร้านขายสัตว์เลี้ยงออนไลน์ แต่เฉพาะทางด้วง - มีทั้งตัวด้วงเป็น ๆ หนอน ดักแด้ ไปจนถึงบ้านด้วง ดิน อาหาร

### ทำไมต้องแยกสินค้าเป็น 2 กลุ่ม?

| สินค้าสิ่งมีชีวิต | สินค้าทั่วไป |
|---|---|
| ตัวด้วง หนอน ดักแด้ | บ้านด้วง ดิน อาหาร |
| ต้องบอกอุณหภูมิ ความชื้น | ไม่ต้องบอก |
| ต้องระวังการขนส่ง | ขนส่งปกติ |
| มีช่วงวัย (larva → pupa → adult) | ไม่มี |
| อาจตายระหว่างขนส่ง | ไม่ตาย |

## 👥 ใครใช้ระบบบ้าง?

### 1. ลูกค้าทั่วไป (Customer)

**สิ่งที่ทำได้:**
1. เข้าเว็บ → ดูสินค้า → กดเข้าไปดูรายละเอียด
2. เพิ่มสินค้าลงตะกร้า (เหมือน Shopee/Lazada)
3. กรอกที่อยู่จัดส่ง → สั่งซื้อ → ได้เลข order
4. ติดตามสถานะ order (รอชำระ → ชำระแล้ว → จัดส่งแล้ว → ได้ของ)

**จุดสำคัญ:**
- ไม่จำเป็นต้องสมัครสมาชิก (guest checkout)
- ถ้าสมัครสมาชิก ประวัติ order จะถูกเก็บไว้
- ดู order ได้โดยใช้ "เลข order + เบอร์โทร/อีเมล" (กันคนอื่นมาดูของเรา)

### 2. ผู้ดูแลร้าน (Admin)

**สิ่งที่ทำได้:**
1. เข้าสู่ระบบผ่านหน้า `/admin/login`
2. จัดการสินค้า: เพิ่ม แก้ไข ลบ (จริง ๆ คือ archive)
3. ดูรายการ order ทั้งหมด
4. เปลี่ยนสถานะ order (เตรียมของ → จัดส่ง → ส่งสำเร็จ)
5. ใส่เลข tracking ขนส่ง
6. ดู dashboard สรุปยอดขาย

## 🔄 Flow การซื้อขาย (สำคัญ!)

```
[ลูกค้า] ดูสินค้า → เพิ่มตะกร้า → Checkout → สร้าง Order
                                                           ↓
[ระบบ] ตัด stock ชั่วคราว (reserve) ← รอชำระเงิน 24 ชม.
                                                           ↓
[ลูกค้า] ชำระเงิน (mock ตอนนี้) → ระบบ lock stock → สถานะ "paid"
                                                           ↓
[แอดมิน] เห็น order ใหม่ → เตรียมสินค้า → ใส่เลข tracking → ส่งของ
                                                           ↓
[ลูกค้า] ติดตาม order ได้ → เห็นว่าส่งแล้ว → ได้รับสินค้า
```

### ถ้าลูกค้าไม่ชำระเงินภายใน 24 ชม.?

ระบบจะ **ยกเลิก order อัตโนมัติ** และ **คืน stock** ให้สินค้าคนอื่นซื้อได้

### ถ้าแอดมินยกเลิก order?

stock จะถูกคืนอัตโนมัติเช่นกัน

## 💰 เรื่องเงิน (สำคัญมาก)

### หน่วยเงิน
- ใน database เก็บเป็น **สตางค์** (integer) ไม่ใช่บาท
- ตัวอย่าง: ราคา ฿850 → เก็บเป็น `85000` (850 บาท × 100)
- ทำไม? ป้องกันปัญหาทศนิยม float (0.1 + 0.2 ≠ 0.3)

### ใครคำนวณราคา?
- **Backend เท่านั้น** ที่คำนวณยอดรวมจริง
- Frontend แสดงราคาเพื่อความสะดวก แต่ backend ต้องคำนวณใหม่ทุกครั้งก่อนสร้าง order
- ป้องกันลูกค้า hack ราคาจาก browser

### คูปองส่วนลด
- มี 2 แบบ: ลดเป็นเงินตายตัว (฿50) หรือลดเป็น % (10%)
- ใช้คูปองได้ตอน checkout
- ระบบตรวจสอบว่าคูปองยังใช้ได้ไหม (หมดอายุ? ใช้ครบโควต้า?)

## 🔐 ความปลอดภัยเบื้องต้น

### ข้อมูลลูกค้า
- รหัสผ่านถูกเข้ารหัส (hash) ไม่มีใครอ่านได้แม้แอดมิน
- เบอร์โทร/ที่อยู่ ไม่แสดงให้คนอื่นเห็น (แม้จะรู้เลข order)
- ต้องใช้เบอร์โทรหรืออีเมลยืนยันตัวตนก่อนดู order

### การชำระเงิน
- ตอนนี้ใช้ **mock payment** (จำลอง) เพราะยังไม่ต่อ payment provider จริง
- mock payment endpoint จะปิดอัตโนมัติถ้า deploy จริง
- ในอนาคตจะต่อ Omise/Stripe ได้

### Admin
- Admin login ใช้ session token ที่มีวันหมดอายุ 24 ชม.
- ไม่สามารถเดา session ได้ (ไม่ใช่เลขง่าย ๆ)
- ทุก action สำคัญต้อง login ก่อน

## ❓ คำถามที่ถามบ่อย

**Q: ทำไมต้องแยก web กับ api?**  
A: เพื่อความปลอดภัย คำนวณราคา/สต็อกที่ backend เท่านั้น ไม่ให้ลูกค้าเล่นกับ frontend แล้วหลอกระบบได้

**Q: ทำไมใช้ PostgreSQL ไม่ใช้ MySQL?**  
A: PostgreSQL รองรับ transaction ที่ซับซ้อนได้ดีกว่า สำคัญมากกับการตัดสต็อก (กันซื้อซ้ำ)

**Q: ตอนนี้รับเงินจริงได้ยัง?**  
A: ยัง ต้องต่อ payment provider จริง (Omise/Stripe) ใน Phase ต่อไป ตอนนี้เป็นระบบจำลอง

**Q: ขึ้น server จริงยากไหม?**  
A: ไม่ยาก มี Dockerfile และ docker-compose รองรับแล้ว ดู [คู่มือ Deploy](./deployment/README.md)
